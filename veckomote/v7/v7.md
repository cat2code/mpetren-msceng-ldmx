## 4) Diskussion: ML-modell för LDMX (inspirerat av CMS MLPF)

**Mål (thesis)**

* Robust **reconstruction i ECAL under pile-up** + tydliga **metrics** mot baseline.
* Vi har pratat om att **tracker-info** sannolikt är värdefullt (början i ECal är rimlig, men målet är inte att begränsa sig i onödan).

**Insikt från CMS MLPF**

* De gör **clustering först** (calo clusters) + tracking, sedan **global tolkning/linking** med ML. “Clustering-first” verkar vara standard.
* Jag har flowcharts i Miro som jag vill få din input på:

  * [https://miro.com/app/board/uXjVIYc7JN4=/?share_link_id=241999863387](https://miro.com/app/board/uXjVIYc7JN4=/?share_link_id=241999863387)

**Toy model idag**

* Hit-level GNN: nodes = reconstructed hits, modellen lär sig att tilldela hits till cluster A/B/C eller noise.
* Kan ses som en “learned clustering” (potentiellt ersätta rule-based clustering), men är tungt (många hits) och mindre “experiment-lik” pipeline.

### 4.1 Två vägar (för att landa i en tydlig plan)

#### Option 1: Hit-level learned clustering (ersätta CLUE) — *local reconstruction*

* **hits → GNN → (clusters + electron count + CLUE-liknande metrics)**
* Fördelar: kan direkt angripa clustering/denoising under pile-up.
* Risker: hög compute + svårare att motivera vs etablerad pipeline inom HEP.

#### Option 2 (föredrar): Cluster-first + ML refinement — *global reconstruction*

* A: **hits → CLUE/clustering**, B: track, **A+B → ML** (merge/split/refine + classify) → count
* Fördelar: lägre compute, närmare PF/MLPF, mer sannolikt nyttigt för LDMX + tydlig baseline.
* Nackdelar: beroende av upstream cluster quality.

**Fråga till mötet**

* Är **Option 2** rimligast som huvudspår?

  * Jag får fortfarande tydlig baseline (“CLUE/PF-only”) och kan bygga mätbar förbättring ovanpå.
  * Eller är jag i praktiken redan på “Option 2” och tänker bara för långt fram?

### 4.2 Option 2 mer konkret

1. **Baseline + metrics**: count accuracy, merge/split rate, prestanda vs pile-up/hit multiplicity.
2. **Cluster features**: (E_\mathrm{sum}), centroid, shower shapes (RMS), (N_\mathrm{hits}), depth/layer-summaries.
3. **ML på cluster-graph** (nodes=clusters + andra outputs, edges=geometri/likhet; output = merge/split/refine eller direkt count).
4. **Tracker context (stretch eller in-scope?)**

   * Ex: lägga till tracker-noder + attention/meddelandepassning mellan tracker och clusters.

### 4.3 Diskussionspunkter

* Baseline: **CLUE**, **PF**, eller båda?
